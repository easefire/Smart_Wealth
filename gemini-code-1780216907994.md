# Smart Risk · 金融风控与智能运营中台

> 基于 Spring Cloud 与 Spring AI 构建的企业级微服务中台。作为《Smart Wealth》理财平台的**旁路风控与智能运营系统**，融合 **多维实时流控**、**可配置风控决策引擎**、**Python 算法打分** 与 **大模型 Agent**，解决金融高频交易下的防刷/反欺诈痛点，并实现内部合规运营的人机协同自动化。

## 一、定位与业务痛点（双子星架构）

本项目不直接处理资金，而是作为《Smart Wealth》的**“安全护盾”与“数据大脑”**，通过 Nacos/OpenFeign 与主交易系统对接，实现**主交易链路（强一致）**与**旁路风控/运营链路（高吞吐、可降级）**的解耦。

**要解决的真实痛点：**
- **羊毛党/脚本刷单**：同一用户/IP/设备在短时间内高频申购赎回、薅活动收益。
- **异常资金行为**：大额拆分、快进快出、关联团伙协同操作。
- **合规运营效率低**：反洗钱规则查询、对账问数、违规账户处置全靠人工。

| 维度 | 主项目：《Smart Wealth》 | 本项目：《Smart Risk》 | 协作链路 |
| :--- | :--- | :--- | :--- |
| **定位** | 核心账务与交易执行 (OLTP) | 实时风控拦截 + 智能运营 (风控 / AI) | 风控拦截恶意流量，保护账务基座 |
| **语言栈** | 纯 Java (JDK 17) | Java + Python (FastAPI) | 跨语言微服务，兼顾并发与模型运算 |
| **数据源** | MySQL 主库 | Redis（频次/名单/向量）+ MySQL 只读账号 | Agent 问数走只读账号，读写隔离 |
| **拦截点（前置）** | 抛出申购/赎回请求 | 多维滑窗限频 → 决策引擎 → 模型打分 | 不通过则**同步阻断**交易 |
| **处置点（后置）** | 提供 `freezeUserAccount` 内部 API | Agent/规则判定违规 → 生成工单 | 人工复核确认后**回调冻结** |

> 接入方式复用 Smart Wealth 已预留的内部接口通道（`/sw/system/**` + IP 白名单过滤器），风控侧通过内部 token 调用，无需改造主链路鉴权。

## 二、项目技术栈

* **后端基座**：Spring Boot 3 + Spring Cloud（Nacos 注册/配置中心、Spring Cloud Gateway、OpenFeign）
* **韧性治理**：Resilience4j（超时、熔断、舱壁、限流）
* **风控决策**：可配置规则引擎（Aviator / QLExpress 表达式 + Nacos 热更新）
* **高频缓存与限流**：Redis（ZSet 滑动窗口、布隆过滤器名单、Lua 原子脚本）
* **智能 AI 层**：Spring AI（OpenAI 兼容 API，预留本地私有化模型）
* **向量检索**：Redis Vector（备选 pgvector / Milvus，按数据规模选型）
* **算法模型层**：Python + FastAPI + scikit-learn（LogisticRegression / XGBoost）
* **消息解耦**：RabbitMQ（事后风控削峰、样本回流、工单事件）
* **可观测性**：Micrometer + Prometheus + Grafana；决策全链路可回溯审计

## 三、核心架构与业务链路

风控分为**同步强阻断**（毫秒级、在交易关键路径上）与**异步事后扫描**（秒/分钟级、不阻塞交易）两条链路，再叠加**智能合规 Agent**。

### 3.1 实时风控防刷链路（同步强阻断，毫秒级）

1. **流量接入**：上游（Gateway / 交易服务）携带 userId、IP、设备指纹、金额等特征发起风控校验。
2. **多维滑窗限频（Java + Redis）**：对 **用户 / IP / 设备** 多个维度分别用 Redis `ZSet` 滑动窗口计数，`ZREMRANGEBYSCORE` 清理过期、`ZCARD` 取窗口计数，Lua 脚本保证“写入+清理+计数”原子性，防御高频重放。
3. **名单前置过滤**：布隆过滤器拦黑名单（O(1)、省内存），白名单直接放行，灰名单转人工/加严策略。
4. **决策引擎裁决**：规则引擎 + 模型打分加权融合，输出 `放行 / 加验 / 阻断`（详见第四章）。
5. **模型打分（Java → Python）**：用 OpenFeign **同步调用** Python 打分服务，返回 0~1 欺诈概率。
6. **降级熔断（关键）**：Resilience4j 分别配置 **200ms 超时** 与 **慢调用比例/失败率熔断**（二者是不同机制）。Python 抖动或熔断打开时，**降级为纯规则裁决 + 放行小额**，确保“风控故障绝不拖垮核心交易”。

### 3.2 事后风控扫描链路（异步削峰，秒~分钟级）

> 不是所有风险都要卡在交易关键路径上。把“可以慢一点”的判定挪到异步链路，既降低主链路延迟，又能跑更重的模型/关联分析。

1. 交易事件经 **RabbitMQ** 投递到事后风控消费者（削峰、可堆积、可重试）。
2. 跑更重的特征聚合与模型（如时间序列、行为序列异常检测）。
3. 命中风险 → 生成处置工单（不直接冻结），进入人机协同流程（见 3.3 / 4.4）。

### 3.3 智能合规 Agent 链路（人机协同）

1. **知识基座（RAG）**：将《反洗钱合规手册》等文档切片、Embedding 入向量库；System Prompt 严格约束“仅依据检索到的上下文回答”，相似度低于**经调参确定的阈值**时拒答，从源头压制幻觉。
2. **动态问数（Text2SQL）**：注入只读表结构 DDL，管理员自然语言提问 → LLM 生成 SQL → 在**数据库只读账号**上执行。安全三道闸：①库级只读权限；②SQL 解析白名单（仅 `SELECT`、限表限字段、禁子查询写）；③强制 `LIMIT` + 查询超时。
3. **动作执行（Tool Calling）**：注册 `freezeUserAccountTool` 等工具。LLM 判定高危时**不直接执行**，而是降级为 `PENDING` 工单，经合规人员一键确认后才回调 Smart Wealth 内部接口冻结账户（人机协同状态机，见 4.4）。

## 四、风控决策引擎设计

> 这是本项目区别于“玩具 demo”的核心：不把风险逻辑写死在代码里，而是做成**可配置、可热更新、可灰度**的决策中枢。

### 4.1 分层裁决
```
请求特征
  │
  ├─ ① 名单层   黑/白/灰名单（布隆过滤器 + Redis）   → 命中黑名单直接阻断
  ├─ ② 规则层   表达式规则集（频次/金额/时段/地域）  → 命中高危规则阻断/加验
  ├─ ③ 模型层   Python 欺诈概率打分                  → 概率加权进总分
  │
  └─ 融合裁决  规则分 × w1 + 模型分 × w2 → 阈值映射  → 放行 / 加验(二次校验) / 阻断
```

### 4.2 规则可配置 + 热更新
- 规则用表达式（Aviator / QLExpress）描述，如 `freq_1min > 10 && amount > 50000`，**存 Nacos 配置中心**，修改后无需发版即时生效。
- 支持规则分级（提示/加验/阻断）与命中权重，便于运营按风险偏好调参。

### 4.3 影子模式（Shadow Mode）/ 灰度
- 新规则或新模型先**影子运行**：照常计算结果但**不真正阻断**，只记录“如果生效会怎样”。
- 对比影子决策与线上决策的拦截率/误杀率，验证无误后再切为生效。**避免新规则上线直接误伤真实用户。**

### 4.4 处置工单状态机（人机协同）
```
风险命中 → 生成工单(PENDING) → 合规人员复核 → APPROVED → 回调冻结账户 → DONE
                                          └→ REJECTED → 关闭(误报，回流为负样本)
```
- 所有“写操作/高危处置”一律先落工单，杜绝模型/LLM 直接动账户的风险。
- 复核结论**回流为训练样本**，形成数据闭环（见第七章）。

## 五、核心技术亮点（面试防守点）

### 5.1 跨语言微服务治理与可降级高可用
使用 Nacos 统一纳管 Java 与 Python 节点，在风控调用链路深度植入 **超时 + 熔断 + 舱壁隔离**（Resilience4j）。即使 Python 打分节点宕机，主交易链路也能通过**降级为纯规则裁决**平稳运行。
> 防守要点：超时（单次请求 200ms 上限）与熔断（基于失败率/慢调用比例的滑动统计）是**两个独立机制**，分别配置、协同生效。

### 5.2 基于 Redis 的多维高性能限流引擎
摒弃数据库 `COUNT()` 查库方案，针对高频 API 请求，设计基于 Redis `ZSet` 的滑动窗口算法：`ZREMRANGEBYSCORE` 实现过期窗口 O(log N) 清理，Lua 脚本保证原子性，对 **用户/IP/设备** 多维并行限频。
> 可复现指标：本机 JMeter 压测下，限流判定单节点稳定支撑数千 QPS（**以实际压测数据为准，不写无依据的可用性百分比**）。

### 5.3 可配置、可灰度的风控决策引擎
规则表达式化 + Nacos 热更新 + 影子模式灰度，使风控策略具备**“不发版即可调整、上线前可验证误杀率”**的工程能力，而非把阈值硬编码在代码里。

### 5.4 企业级防幻觉 RAG 架构
System Prompt 严格设定边界（仅依据检索上下文作答）+ 相似度阈值拦截（阈值经调参确定，而非拍脑袋写死），确保合规问答可回溯至原始文档，显著降低 LLM 幻觉。

### 5.5 闭环的 Function Calling 安全设计
为解决 LLM 直接写库的隐患，所有高危写操作（如冻结账户）被拦截降级为 `PENDING` 工单，经人工确认后释放执行，兼顾 AI 效率与金融级严谨。

## 六、智能合规 Agent 演进路线

遵循逐步迭代，便于独立调试与问题溯源：
* **Level 1（RAG）**：文档切块 → Embedding → 向量检索增强问答。（解决：合规规则查询）
* **Level 2（Text2SQL）**：注入只读 DDL → 自然语言转 SQL → 只读账号执行 → 数据总结。（解决：运营对账问数）
* **Level 3（Tool Calling）**：注册工具 → LLM 决策 → 工单拦截 → 人工确认后跨服务回调。（解决：违规处置自动化）
* **Level 4（进阶·可选）**：引入 ReAct/多步推理与短期记忆，结合 Guardrails 输出校验与离线 Eval 评测集，对 Agent 决策做质量度量；用**语义缓存**降低重复问答的 token 成本。

## 七、可观测性与工程化闭环

> 风控系统“看不见就等于没有”，工程化与可观测性是它能上生产的前提。

- **决策可回溯**：每次裁决落审计日志（命中的规则、模型分、最终动作、耗时），支持事后复盘与申诉。
- **指标监控**：Micrometer 暴露指标 → Prometheus 采集 → Grafana 看板（QPS、P99 决策延迟、各规则命中率、熔断状态、误杀率）。
- **样本闭环**：工单复核结论（确认违规 / 误报）**回流**为正/负样本，定期重训模型，模型效果可持续迭代。
- **数据隔离**：风控读 Smart Wealth 数据走**只读账号**，绝不写主库；自身风险数据独立存储。

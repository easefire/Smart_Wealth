## 一、与《Smart Wealth》的双子星架构图谱

本项目不直接处理资金，而是作为**“安全护盾”**与**“数据大脑”**，通过 Nacos/OpenFeign 与第一项目《Smart Wealth》无缝咬合。这种设计彻底实现了**主交易链路（强一致性）**与**旁路审计/运营链路（高吞吐、高可用）**的架构解耦。

| 维度 | 第一项目：《Smart Wealth》 (主干) | 第二项目：《Smart Risk》 (中台) | 协作链路说明 |
| :--- | :--- | :--- | :--- |
| **定位** | 核心账务与交易执行 (OLTP) | 实时安全拦截与智能运营 (OLAP / AI) | 风控中台拦截恶意流量，保护账务基座不被击穿。 |
| **语言栈** | 纯 Java (JDK 17) | Java + Python (FastAPI) | 跨语言微服务调用，兼顾并发处理与复杂模型运算。 |
| **数据源** | MySQL (读写主库) | Redis (频次/向量) + MySQL (只读从库) | Agent 查询对账数据时强制走从库，实现读写隔离。 |
| **交互点 1** | 抛出用户申购/赎回请求 | 接收请求 -> ZSet 滑动窗口查频 -> Python 打分 | **前置熔断**：风控评估不通过，直接阻断交易。 |
| **交互点 2** | 提供 `lockUser()` 等高危 API | LLM 判定违规 -> 触发 Function Calling | **后置执行**：Agent 代替人工，跨微服务发起账户冻结。 |

## 三、核心架构与业务链路（一步步拆解）

项目开发与链路流转严格遵循“由简入深、模块化解耦”的逻辑，划分为两大数据流：

### 3.1 实时风控防刷链路（毫秒级响应）

1. **流量接入**：上游（网关或交易模块）发起 HTTP/MQ 请求，携带用户 ID、IP、金额等特征。
2. **高频校验（Java + Redis）**：通过 Redis `ZSet` 实现滑动时间窗口，判定该用户近 1 分钟内的交易频次是否异常（例如防御羊毛党的高频重放攻击）。
3. **模型评分（Java RPC -> Python）**：利用 `OpenFeign` 异步调用 Python 侧的微服务接口。Python 端加载特征评估逻辑，返回 0.0~1.0 的欺诈概率（Fraud Score）。
4. **降级熔断**：配置 Resilience4j 断路器。当 Python 服务网络抖动或超时（> 200ms）时，Java 端触发熔断，默认放行小额交易，确保**“风控系统故障绝对不拖垮核心交易”**。

### 3.2 智能合规 Agent 链路（人机协同）

1. **知识基座（RAG）**：通过 `TokenTextSplitter` 将《反洗钱合规手册》等非结构化文档切片，经 Embedding 转化为向量存入 Redis Vector，解决 LLM 幻觉问题。
2. **动态对账（Text2SQL）**：向 LLM 注入《Smart Wealth》的只读表结构（DDL）。管理员提问“昨天有哪些清算失败的订单”时，LLM 自动生成 SQL 并由 Java 在从库执行，返回数据洞察报告。
3. **动作执行（Tool Calling）**：注入工具函数。若 LLM 综合判断某账户存在高危风险，直接回调系统暴露的 `freezeUserAccountTool`，通过 RPC 跨微服务冻结目标账户，并生成审计工单待人工复核。

## 四、核心技术亮点（面试防守点）

### 4.1 跨语言微服务治理与极致高可用
打破单一语言局限，使用 Nacos 注册中心统一纳管 Java 与 Python 节点。针对金融级场景，在微服务通信链路中深度植入**超时控制与断路器模式**。在压力测试下，即使风控算法节点宕机，主交易链路仍能通过熔断降级策略平稳运行，系统整体可用性达 99.99%。

### 4.2 基于 Redis 的高性能时序指标引擎
摒弃传统的数据库 `COUNT()` 查库方案，针对连续高频的 API 请求（类 50Hz 数据流处理经验移植），设计了基于 Redis `ZSet` 的滑动时间窗口算法。利用 `ZREMRANGEBYSCORE` 实现过期窗口数据的 $O(\log N)$ 级快速清理，单节点轻松支撑数千 QPS 的频次拦截。

### 4.3 企业级防幻觉 RAG 架构落地
拒绝“玩具级”大模型对话。系统通过 System Prompt 严格设定边界条件（System Message: You must ONLY use the provided context...），结合向量相似度阈值拦截（Similarity Score < 0.8 则拒绝回答），确保金融合规问题的输出 100% 可回溯至原始产品说明书，将 LLM 幻觉率降至极低。

### 4.4 闭环的 Function Calling 安全设计
不仅仅停留在“信息查询”，更赋予 AI 实体业务操作能力。为解决 LLM 直接写库的安全隐患，设计了**“工单流转与人机协同”**状态机：LLM 发起的写操作（如冻结账户）被拦截并降级为 `PENDING` 状态，待合规人员一键点击确认后方可释放执行，兼顾了 AI 的高效与金融级的严谨。

## 五、AI Agent 演进路线

项目在开发过程中，严格遵循了逐步迭代的步骤，方便独立调试与问题溯源：
* **Level 1 (RAG)**：文档读取 -> 切块 (Chunking) -> 向量化 (Embedding) -> 检索增强问答。（**解决：金融规则查询痛点**）
* **Level 2 (Text2SQL)**：注入 DDL -> 自然语言转 SQL -> 查只读库 -> 数据总结。（**解决：日常营运对账痛点**）
* **Level 3 (Tool Calling)**：注册工具 -> LLM 决策树判定 -> 触发跨服务 RPC -> 拦截器校验。（**解决：违规处理自动化痛点**）
